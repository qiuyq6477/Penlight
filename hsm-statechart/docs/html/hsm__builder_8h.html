Detailed Description

Declarative interface for defining states.

The builder is a layer on top of the core hsm statemachine code. The builder is a layer on top of the core hsm statemachine code. It is not necessary to use, or even include, the builder to use hsm-statechart.

Definition at line 955 of file hsm_builder.c.

int hsmState ( const char *  name)

Declare a new state.

Parameters:
nameName relative to enclosing state; if no enclosing state, assumed to be the top state of a machine.
Returns:
int opaque handle to an internal name

This function is idempotent(!)

If you call it with the same name, in the same begin/end scope, in the same run of the app, you will get the same value back out. The value is not gaurenteed to be the same across different launches of the same application.

Names are combined using "::" as per the statechart spec. ex.: Parent::Child::Leaf

What this all means is you cannot have two top level states both called "Foo" in your application. You can however reuse states by within a statemachine provided they have different parents: ( ie. { Outer: { InnerA: Foo } { InnerB: Foo } } is okay )

note: i actually tried pre-allocating hsmState objects and storing those in hsmGoto, but if the user code is using string names, the allocation has to know how/whether to copy the string; that's not always obvious until hsmBegin()
